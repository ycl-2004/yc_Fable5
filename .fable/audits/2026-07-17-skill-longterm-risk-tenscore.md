## 上下文成本三层模型（回答「会不会无限变长」）

| 层 | 何时付费 | 现值 | 膨胀后果 | 现状风险 |
|----|---------|------|---------|---------|
| 常驻层（CLAUDE.md 段 + description） | 每个会话 | ~700 tok | 全局税，最贵 | 低——但 description 增触发词会加税 |
| 触发层（SKILL.md 正文） | 复杂任务加载时 | ~2500 tok | 每次复杂任务付 | 低——v1.4 仅 +2 行，克制 |
| 按需层（references/examples） | 用到才读单文件 | 单次 1-3 文件 | 局部、可控 | 中——复盘规则的必然沉积地 |
| 磁盘层（.fable/audits/、task.md） | 从不进 context | — | 无 token 后果 | 无 |

**结论：不会「无限变长」地伤害每次对话**——架构分层正确，最能涨的（审计归档）不进 context，最贵的（常驻层）最难被动增长。真实的膨胀路径是按需层被复盘规则慢慢灌厚 + CHANGELOG 永久累积 + skill-tool-map 随生态线性增长。

## 核心洞察：两个诉求同解

「怕无限变长」与「要完整写下最强能力」看似矛盾（写得越全越长），实为同一个解：Fable 真实的思考方式不是枚举（每失误 +1 规则、每任务型 +1 工作流——枚举必然膨胀），而是**少数原则 + 现场判断**。进化方向定为「**规则数量守恒或递减、抽象层级递升**」——新失误优先归并改写已有规则，新增是最后手段；同类规则定期压缩成更高阶原则。这同时治膨胀、也才真正逼近「最强思考方式」。

## 整改项（做什么就能过，非批评）

1. **规则守恒条款**（治 #3/#5）：evolution.md 复盘三问后加第四问「已有规则能否覆盖或合并？优先改写，新增是最后手段」；大版本例行一次「合并同类 + 活水退役（已被运行时内建的降级出注入路径）」审查。
2. **CHANGELOG 滚动归档**（治 #3）：evolution.md 只留最近 2-3 版，更早移入 HISTORY.md（脱离按需读取路径）。
3. **行数软预算**（治 #3）：单 reference ~150 行软上限，超限触发压缩而非分裂新文件；写进保鲜例行。
4. **思考内核层**（治 #10）：在 FABLE_CORE 或新 reference 补 4 条原则——先测量再断言、权衡显性化、反枚举压缩、打破规则的条件。原则不随任务数增长，本身就是防膨胀的。
5. **Codex 服从率实测**（治 #7）：已在未做清单，优先级应上调——契约零实证是可移植性最大虚位。
