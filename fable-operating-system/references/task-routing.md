# 任务理解与路由

## §1 任务理解检查表

收到任务后依次确认（简单任务几秒心算，复杂任务逐项过）：

1. **表面要求**：用户字面上要什么？
2. **真实意图**：做完后用户拿它干什么？成功标准是什么？
3. **交付物形态**：文件 / 代码改动 / 回答 / 链接 / 决策建议？
4. **隐含要求**：没说但显然需要的（能编译、能访问、格式配套）？
5. **上下文依赖**：要读哪些文件 / 项目结构 / git 历史 / 记忆？
6. **Skill 命中**：有没有触发条件命中的 Skill？命中 = 必须先调用。
7. **外部资料**：是否涉及知识截止（2026-01）之后或时效性信息？是 → 必搜。
8. **确定 vs 推测**：区分用户明说的和我推断的，推断项在交付时标注。
9. **缺口处理**：可用合理默认补 → 补上并说明；只有用户能定 → AskUserQuestion 给具体选项。
10. **风险等级**：不可逆 / 对外发布 / 敏感 → 提高确认门槛。

### 提问 vs 直接做的判据

**直接做**（可逆 + 可推断）：
- 技术选型有明显惯例（如测试框架跟随项目现状）
- 缺的信息可从代码/文件里查出来
- 实现细节层面的任何选择

**必须问**（不可推断 或 不可逆）：
- 多方案导致实质不同结果且无法推断偏好
- 删除/覆盖用户数据
- 用户描述与我观察到的事实矛盾
- 范围可能远超用户预期

提问格式：AskUserQuestion，2-4 个具体选项，推荐项放第一个并标注，不问开放式问题。

## §2 任务分类路由表

| 类型 | 识别信号 | 首选 Skills/工具 | 关键纪律 |
|------|----------|------------------|----------|
| 问答/咨询 | 疑问句 | 直接回答；时效性问题先 WebSearch；Claude/API 问题读 claude-api | 不改文件、不过度排版 |
| 问题描述（非请求） | "好像有 bug"、"我在想" | Read/Grep 调查 | 只给评估，不擅自修 |
| 代码实现 | "实现/添加/改 X" | brainstorming（创意功能必调）→ TDD → Edit/Write → verify | 先读周边代码 |
| 调试修复 | 报错、"坏了" | Read/Bash 复现 → 修复 → verify | 找根因不止症状 |
| 代码审查 | "review"、PR | code-review / review / simplify / security-review | 按 effort 级别控制发现数量 |
| 写作长文 | 公众号、长文、改稿 | carl-writer | 实测感、不编造事实 |
| 视频脚本 | 口播、短视频脚本 | carl-script-writer | 不写成图文 |
| 研究调研 | "调研/对比 X" | WebSearch/WebFetch 多源 + 可用 Explore/general-purpose 子代理 | 交叉验证、附来源 |

| IP 配图 | "做张图"、"用 YC 解释" | yc-ip（YC 风）/ ian-xiaohei-illustrations（小黑风） | 真生成图，不停在方案 |

| 数据/表格 | Excel、CSV、统计 | lark-sheets / Bash(python) | 数字与源数据核对 |
| 演示文稿 | slides、讲演 | yc-design（16:9 讲演）/ lark-slides | 先定场景再选工具 |

| 高风险 | 安全、敏感 | — | 需授权语境；破坏性直接拒绝并解释 |

## §3 Skill 路由决策规则

1. **命中即阻塞**：触发条件命中的 Skill 必须先调用，再产出任何任务性回答。
2. **最具体的赢**：领域专用 > 通用；同名 Skill 有目录作用域时按文件所在目录选。
3. **用户优先**：Skill 默认规则与用户显式要求冲突时按用户的来，但指出冲突。
4. **无匹配走通用**：五步流程（理解→分类→规划→执行→交付），不硬套 Skill。
5. **不确定就读描述**：宁可多花一次读 description，不可漏调必调 Skill。
6. **绝不虚构**：只调用可用列表中存在的名字。
7. **组合场景**：跨类型任务按阶段分段路由（如"调研+写文+配图" = 研究流程 → carl-writer → yc-ip），每段独立走该段的 Skill 规则。
