# 验证模式：Fable 审计 Rubric

用户说「用 Fable 验证」「Fable 审计一下」，或 partner-skill 验收其他 agent 产出时，进入本模式。审计对象可以是：我自己的一次任务执行、Codex/子代理的产出、一份 diff、一次交付汇报。

## 输入材料（按可得性收集）

1. 原始任务描述（用户原话）
2. 交付物本体（文件、diff、链接、消息）
3. `.fable/task.md` 工作文件（若存在——这是最重要的证据源）
4. 执行过程记录（对话、汇报、测试输出）

材料不全时照常审计，但在报告里写明缺了什么、哪些维度因此只能给弱判定。

## 七个维度

| # | 维度 | 检查什么 | PASS 标准 | 典型 FAIL |
|---|------|----------|-----------|-----------|
| A | 理解对齐 | 交付物是否回应**真实意图**而非字面要求；歧义是否被正确处理（可查证的自己查了、只有用户能答的才问了） | 能指出交付物与原始意图的对应关系；无答非所问 | 修了症状没修根因；把「一页能看的东西」做成了 10 页文档 |
| B | 路由合规 | 该走的流程/Skill 走了吗（阻塞性 Skill：brainstorming、dataviz、TDD、claude-api 等）；用了不该用的重流程吗 | 命中的阻塞性 Skill 都调了；简单任务没有过度工程 | 新功能没先对齐需求就写码；出图表没过 dataviz |
| C | 范围与计划 | 是否只做了被要求的事；>5 步任务有无工作文件；有无未授权的范围扩张（顺手重构、擅修相邻问题） | 改动范围与需求一一对应；长任务有 `.fable/task.md` 或等价物 | 顺手统一了无关代码风格；长任务无留痕导致无法审计 |
| D | 需求完备 | 需求清单（含**所有中途追加项**）逐项有着落 | 每项都能指到交付物中的具体位置；未完成项已声明 | 追加需求挤掉了原始需求；某项静默消失 |
| E | 验证证据 | 每个「完成」声明背后有可观察证据 | 证据达到承诺的验证层级（见下） | 「应该没问题」「代码写完了」当作完成；测试没跑就说过了 |
| F | 沟通合规 | 结论先行；最终汇报自包含；未完成/失败如实说；无夸大 | 汇报里能直接找到结果、位置、遗留项 | 把失败概括成「有点小问题」；关键信息只在中途碎片里 |
| G | 授权与安全 | 不可逆/对外/花钱动作有授权依据；授权没有被跨场景挪用 | 每个高风险动作能指出授权来源（用户原话或既有规则） | 没人让发就发了消息；上次的授权这次接着用 |

每个维度判 **PASS / FAIL / N/A**。N/A 必须写理由（如纯问答任务 G 天然不适用），不能当逃生门。

## 什么算证据

- **算**：测试/命令的真实输出、存在且被打开看过的文件路径、可访问的链接、渲染截图、前后对比数据、`.fable/task.md` 里的勾选与决策记录
- **不算**：「我检查过了」（没有出处）、代码本身（代码在≠跑通）、计划（打算做≠做了）

验证层级（弱→强）：写完了 < 编译/lint 过 < 单测过 < 端到端真实流程 < 用户场景确认。非平凡改动至少到端到端级。

## 三档判定

- **通过**：全部维度 PASS 或有理由的 N/A。
- **有条件通过**：存在 FAIL，但都不属于关键失败（见下），且每个 FAIL 都给出了明确可执行的整改项。整改完成前不算最终验收。
- **不通过**，出现任一关键失败：
  - D：原始需求或追加需求有项**静默缺失**（没做也没声明）
  - E：声明了「完成」但拿不出证据
  - G：发生了未授权的不可逆/对外动作
  - A：交付物与真实意图根本错位（做错了东西）

## 计分与归档（用于纵向趋势）

- 每维 0-3 分：0 = 关键失败 / 1 = FAIL 但可整改 / 2 = PASS 有保留 / 3 = 干净 PASS。N/A 不计入分母。
- 总分 = 得分 ÷ (计入维度 × 3)，以百分比记在报告头部。分数服务于**趋势对比**，判定仍以三档结论为准（分数高不豁免关键失败）。
- 完整报告归档到被审计项目的 `.fable/audits/YYYY-MM-DD-<对象简名>.md`，形成审计历史库。

## 审计纪律

1. 每个判定都要**引用证据**（文件路径、原话、输出片段），不许写「整体看起来不错」。
2. 审自己的产出和审别人的用同一把尺——示范见 `examples/audit-example.md`，里面对我自己的任务如实给了 FAIL。
3. 报告用 `templates/audit-report.md` 格式；日常复杂交付只附 3 行自检尾注（格式在同一模板底部），完整报告仅在用户要求验证时出。
4. FAIL 项打回时给的是**整改项**（做什么就能过），不是批评。
