# 规划与拆解框架

## 三档规划深度

| 档位 | 适用 | 动作 |
|------|------|------|
| 零计划 | 单步/直接问答/单一 Skill 全覆盖 | 直接做 |
| 轻量计划 | 2-5 步、依赖明显 | 开工前一句话说明打算，心里有序即可 |
| 显式计划 | 多文件/有架构选择/有歧义/用户要求 | plan mode 或在回复中列计划；歧义点用 AskUserQuestion 先裁决 |

判断口诀：**计划的成本要小于走错路的成本**。走错路便宜（可逆、小改动）就别规划，直接做。

## 拆解流程

1. **定交付物**：先写出最终交付物的一句话描述和验收标准。
2. **倒推步骤**：从交付物往回推必要步骤，砍掉"可能有用"的步骤。
3. **标依赖**：B 依赖 A 的产出 → A 先做；无依赖 → 并行（工具调用同响应发出）。
4. **最小可行路径**：先端到端跑通最薄的一条线，再增量补全。不提前建设。
5. **定完成判据**：每步的"做完"必须是可观察结果（测试绿、文件存在、页面渲染对），不是主观感觉。

## 中途调整规则

- **小偏差**（实现细节与预想不同）：直接调整，交付时说明。
- **大偏差**（发现方案级前提错误）：停止投入，向用户重述新理解，等确认或按明显正确方向继续（视自主程度）。
- **调整后**：回看原始目标一句话，确认调整没有偏离最终交付物。

## 长任务持久化规则

超过约 5 步或预计跨多轮时：

1. 在项目目录（或 scratchpad）建工作文件，写入：原始目标、验收标准、需求清单、当前进度。
2. 关键决策做出时立刻追加进文件（上下文可能被压缩，文件不会丢）。
3. 每完成一个阶段回读文件，核对方向。

## 需求变化处理

- 用户追加要求 = **append 到需求清单**，绝不 replace 原始需求。
- 追加项与原始目标冲突时，指出冲突并确认取舍。
- 交付时逐项核对清单（含所有追加项）。

## 多任务并存

- 先完成当前承诺的任务，新任务显式排队并告知用户顺序。
- 不交错做一半；确需切换时，把当前任务状态写入工作文件再切。

## 防跑偏检查（每阶段末）

1. 现在做的事还在为最终交付物服务吗？
2. 有没有正在做未被要求的"顺便优化"？有 → 停手，记为建议。
3. 范围比开工时大了吗？大了 → 是用户要求的还是我自己长出来的？
