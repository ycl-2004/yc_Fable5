# 各类任务标准工作流

每类按统一模板：**输入检查 / 检索与上下文 / 规划 / Skills与工具 / 执行步骤 / 质量检查 / 交付格式 / 常见错误 / 失败替代**。

---

## 1. 信息查询与研究

- **输入检查**：问题范围明确吗？要事实还是要评估？深度要求（一句话答案 vs 报告）？
- **检索与上下文**：稳定知识可直接答；有争议/专业性强 → 多源检索交叉验证。
- **规划**：深度研究先列子问题清单；独立子查询并行发。
- **Skills/工具**：WebSearch、WebFetch；大范围代码搜索用 Explore 子代理；Claude/LLM 相关先读 claude-api。
- **执行**：拆子问题 → 并行检索 → 冲突信息找第三源 → 综合。
- **质量检查**：来源可靠性、信息时效标注、区分事实与推断。
- **交付**：短问题给散文；报告类落 Markdown 文件。
- **常见错误**：单一来源就下结论；把训练记忆当最新事实。
- **失败替代**：搜索不可用 → 基于已有知识回答并显著标注时效风险。

## 2. 当前资讯与时效性问题

- **输入检查**：涉及 2026-01（知识截止）之后吗？价格/版本/人事/新闻类一律视为时效性。
- **检索**：**必须搜索**，不凭记忆答。
- **Skills/工具**：WebSearch 优先；具体页面 WebFetch。
- **执行**：搜 → 核对日期 → 引用时标注"截至 X 日期"。
- **常见错误**：自信地给出过时答案。
- **失败替代**：说明无法联网验证 + 给出已知的最后状态 + 建议验证渠道。

## 3. 写作、改写与文案

- **输入检查**：平台？受众？字数？语气？以谁的身份发？有无素材？
- **上下文**：读全部给定素材；不编造用户没提供的事实/经历。
- **Skills**：公众号长文 → carl-writer；视频口播 → carl-script-writer；两者都不匹配走通用写作。
- **执行**：对齐方向 → 加载风格规则 → 初稿 → 按风格自查 → 修订。
- **质量检查**：事实有据、语气一致、平台格式（如公众号的截图位）、无占位符。
- **交付**：落 Markdown 文件（默认），给路径；说明关键创作选择。
- **常见错误**：脚本写成图文；替用户虚构"实测经历"。
- **失败替代**：素材不足时列出缺口清单请用户补充，先交付能写的骨架。

## 4. 代码编写

- **输入检查**：需求可测吗？涉及创意/新功能 → **先过 brainstorming**（阻塞）。
- **上下文**：读周边代码、项目约定、CLAUDE.md。
- **规划**：多文件改动列显式计划。
- **Skills**：brainstorming → test-driven-development → 实现 → verify → （可选）code-review/simplify。
- **执行**：先写失败测试 → 最小实现让测试绿 → 重构 → 端到端 verify。
- **质量检查**：见 quality-checks 代码项；风格随周边代码。
- **交付**：改动文件清单 + 验证证据；用户要求才 commit。
- **常见错误**：跳过 TDD/brainstorming；只跑 lint 不跑真实流程；顺手重构无关代码。
- **失败替代**：卡住 2-3 轮 → codex:rescue 二次诊断。

## 5. 代码调试与错误修复

- **输入检查**：拿到完整报错了吗？能复现吗？
- **执行**：复现 → 沿调用链定位根因 → 最小修复 → 验证原报错消失且无新增失败。
- **质量检查**：能一句话说出根因；对照区分本次改动导致 vs 原有失败。
- **交付**：根因 + 修复 + 验证证据。
- **常见错误**：修症状不修根因；改动比必要的大。
- **失败替代**：无法复现 → 加日志/断言缩小范围；仍不行 → 报告排查边界与假设。

## 6. GitHub、Issue、PR 与 CI

- **输入检查**：目标 repo/分支/PR 号；我有没有权限。
- **工具**：`gh` CLI；PR 审查用 review Skill；本地 diff 用 code-review。
- **执行**：读 Issue/PR 上下文 → 本地操作 → commit 规范（含 Co-Authored-By 尾注）→ push/PR 需用户要求。
- **质量检查**：默认分支不直接 commit（先开分支）；PR body 含生成尾注。
- **常见错误**：未经要求就 push；CI 等待用轮询浪费（应一次长间隔检查）。
- **失败替代**：CI 挂了拉日志分析，不盲目 rerun。

## 7. 文件整理与项目重构

- **输入检查**：整理规则是什么？哪些不能动？
- **执行**：**先完整盘点再动手**（ls/tree/du）→ 提出方案（大动作先确认）→ 移动/重命名 → 核对无丢失。
- **质量检查**：文件数前后核对；引用路径更新（代码里的 import、文档里的链接）。
- **常见错误**：删除/覆盖未看内容的文件；破坏隐式依赖。
- **失败替代**：不确定的文件移入 `_review/` 暂存区而不是删除。

## 8. 文档制作

- **输入检查**：读者是谁？给人看还是给 agent 看？放哪（repo/飞书/网页）？
- **Skills**：repo 内 → Markdown 直写；飞书 → lark-doc；网页 → Artifact（先读 artifact-design）。
- **执行**：定结构 → 写 → 检查代码示例可运行、链接有效。
- **常见错误**：文档与代码现状不符；写给作者自己而不是读者。

## 9. PDF 处理

- **输入检查**：几页？要读全文还是特定页？
- **工具**：Read 的 pages 参数（>10 页必须分段，每次 ≤20 页）；生成/转换用 Bash（python 等）。
- **常见错误**：一次读超页数限制；扫描版 PDF 当文本读（应按图像处理）。
- **失败替代**：解析失败 → 转图像逐页读。

## 10. 表格与数据分析

- **输入检查**：数据在哪（本地/飞书/URL）？口径和单位？
- **Skills/工具**：飞书表格 → lark-sheets；本地 CSV/Excel → Bash(python/pandas)；出图前**必读 dataviz**。
- **执行**：读数据 → 清洗 → 计算 → 关键数字与源核对 → 可视化。
- **交付**：结论先行 + 数据文件/图表路径。
- **常见错误**：不核对就引用计算结果；图表配色随手选（漏 dataviz）。

## 11. 演示文稿制作

- **输入检查**：场景（讲演/汇报）？平台（飞书/网页）？页数与比例？
- **Skills**：个人品牌 16:9 讲演 → yc-design；飞书原生 → lark-slides；NotebookLM 产物 → notebooklm studio。
- **执行**：先定大纲（每页一句话）→ 确认 → 逐页制作 → 整体过一遍视觉一致性。
- **常见错误**：没定大纲直接做页；文字密度过高。

## 12. 图片与视觉设计

- **输入检查**：用途（文章配图/社媒/UI）？风格（YC IP/小黑/通用）？比例？
- **Skills**：yc-ip（YC 形象，必须真出图）/ ian-xiaohei-illustrations（小黑）/ ui-ux-pro-max（UI）/ grid-split（切图）/ pencil（.pen 文件）。
- **质量检查**：真实查看生成结果，不以"命令跑完"为完成。
- **常见错误**：停在文字方案不出图；.pen 用 Read 去碰。

## 13. 视频与动态内容

- **输入检查**：要脚本还是要成品？时长？平台？
- **Skills**：脚本 → carl-script-writer；浏览器操作录制 → claude-in-chrome 的 gif_creator；音视频转写 → lark-minutes。
- **常见错误**：视频脚本写成图文腔。

## 14. 自动化与定时任务

- **输入检查**：触发条件（每次动作时/定时/循环）？跨会话吗？
- **路由**："以后每次 X" → update-config（hooks，因为我记不住跨会话行为）；定时云任务 → schedule；会话内轮询 → loop。
- **质量检查**：配置生效验证（触发一次或说明验证法）。
- **常见错误**：把自动行为写进记忆以为能生效。

## 15. 邮件与日历任务

- **输入检查**：哪个体系（飞书/Gmail）？读还是写？
- **Skills**：Gmail MCP / lark-calendar（lark-mail 已下架，飞书邮箱走 lark-openapi-explorer）。
- **纪律**：读随便，**发送/回复/删除先确认**；草稿可以直接建。
- **常见错误**：把"帮我回复"理解成直接发送（应先给草稿）。

## 16. 用户个人知识、偏好与历史上下文

- **输入检查**：MEMORY.md 有相关条目吗？记忆是否过时（文件/flag 还存在吗）？
- **工具**：持久记忆目录 + share-memory（项目级 Claude/Codex 共享）。
- **执行**：用前验证记忆时效 → 任务结束时把新的持久性偏好/教训写入。
- **纪律**：不存 repo 已记录的事；相对日期转绝对日期；错了就删。

## 17. 高风险、敏感或受限任务

- **输入检查**：是否属于授权安全测试/CTF/防御/教学语境？
- **执行**：有授权语境 → 协助并保持在授权范围内；破坏性技术/DoS/大规模针对/恶意规避 → 拒绝并说明原因和边界。
- **纪律**：拒绝时不提供替代实现路径；模糊地带按保守方向处理。
- **交付**：说清什么能做、什么不能做、为什么。
