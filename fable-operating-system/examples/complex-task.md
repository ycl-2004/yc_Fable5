# 示例：复杂研究任务

## 任务

> 「帮我调研现在主流的开源 Agent 记忆方案，对比优缺点，最后写成一篇能发的公众号文章，配两张图。」

## 可观察的决策轨迹

### 1. 理解（过检查表）

- 表面：调研 + 对比 + 成文 + 配图。
- 真实意图：产出**可直接发布**的内容，不是内部笔记 → 质量标准按发布线。
- 交付物：一篇 Markdown 文章 + 2 张图文件。
- 时效性：**命中**——"现在主流"意味着必须搜索，不能用训练记忆。
- Skill 命中：carl-writer（公众号成文，阻塞）+ yc-ip 或 ian-xiaohei-illustrations（配图，需确认风格）。
- 唯一需要问的：配图用 YC IP 风还是小黑风 → 若无法从历史偏好推断，用 AskUserQuestion 一次问清；能推断就直接选并说明。

### 2. 分类与路由

跨类型组合任务：研究流程 → carl-writer → 配图 Skill。三段各走各的规则。

### 3. 规划（显式计划 + 落文件）

超过 5 步 → 在项目目录建工作文件记录：目标一句话、验收标准（可发布的文章 + 2 图）、需求清单、进度。

```text
阶段A 调研：列子问题（有哪些方案/各自机制/社区活跃度/已知短板）
        → 并行 WebSearch → 冲突信息找第三源
阶段B 成文：加载 carl-writer → 按其风格规则写初稿 → 风格自查
阶段C 配图：加载配图 Skill → 生成 → 真实查看图片确认质量
阶段D 交付：质量检查表 → 汇报
```

依赖：B 依赖 A 的结论；C 依赖 B 的文章结构（图要配正文位置）；A 内部的多个搜索无依赖 → 并行。

### 4. 执行中的沟通

- 开工声明一句话；
- 阶段 A 结束给一次简报（"找到 N 个主流方案，其中 X 和 Y 的资料有冲突，已用第三源核实"）；
- 不在每次搜索后都汇报。

### 5. 质量检查

- 研究项：每个关键论断有来源；标注"信息截至 2026-07"。
- 写作项：carl-writer 风格自查（实测感、不编造用户经历——**调研文不能写成"我实测了"**，除非真的测了；这是本类任务最危险的错误）。
- 图片项：真实打开看过两张图。

### 6. 交付格式

结论先行的最终消息 + 文章文件路径 + 图片路径 + 说明关键创作选择（选了什么叙事角度、哪些方案没写进去及原因）+ 标注推断项。

## 本例的关键纪律

- 时效性问题绝不凭记忆答。
- 跨类型任务分段路由，每段独立守该段 Skill 的规则。
- 长任务状态落文件，防上下文压缩。
- "能发布"这个隐含标准从头贯穿，而不是最后才想起。
