# 示例：代码调试修复任务

## 任务

> 「跑 `npm test` 有三个用例挂了，帮我修一下。」

## 可观察的决策轨迹

### 1. 理解

- 表面：让三个测试变绿。
- 真实意图：修掉导致失败的**缺陷**——如果失败是测试本身写错，修测试；如果是产品代码坏了，修代码。不能为了绿而绿（如删断言）。
- 交付物：代码改动 + 全绿的测试输出。
- Skill 命中：这是 bugfix → test-driven-development 适用（已有失败测试，相当于 TDD 的红灯已就位）；修完 → verify。

### 2. 执行步骤

1. **复现**：先真的跑 `npm test`，拿到三个失败的原始输出——不基于用户转述工作。
2. **对照**：`git status` / `git log` 看最近改动，判断失败是新引入的还是历史遗留。
3. **定位根因**：沿失败断言读被测代码调用链，直到能一句话说出"为什么挂"。三个失败先判断是否同一根因（常见）。
4. **最小修复**：只改与根因相关的代码，不顺手重构。
5. **验证**：重跑完整测试套件（不只跑那三个）——确认原失败消失且**无新增失败**。
6. **verify**：若改的是产品代码，端到端驱动一次受影响的真实流程。

### 3. 沟通

- 开工："先复现这三个失败，再定位根因。"
- 定位到根因时简报一句（这是关键事实）。
- 最终汇报模板：
  1. 结论：三个失败均已修复，全套件 N 通过 0 失败。
  2. 根因：一句话（如 `parseDate` 在 `utils/date.ts:42` 没处理时区偏移，三个用例都栽在这）。
  3. 改动：文件清单 + 每处为什么。
  4. 证据：测试输出摘要。
  5. 边界：如有历史遗留的其他失败——报告但没修，说明原因。

## 本例的关键纪律

- 不基于转述调试，一切从复现开始。
- 修根因不修症状；绝不用删断言/skip 让测试"变绿"。
- 跑**完整**套件防回归。
- 发现的相邻问题（历史遗留失败）报告不擅修。

## 失败替代

无法本地复现（环境差异）→ 报告排查边界 + 建议提供 CI 日志或环境信息；卡住 2-3 轮 → codex:rescue 二次诊断。
