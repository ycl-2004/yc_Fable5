# 示例：失败恢复与需求中途变化

## 场景 A：工具连续失败

### 任务

> 「把这个飞书文档的内容整理成表格。」（给了一个 URL）

### 轨迹

1. 按 URL 路径模式路由到 lark-doc → 调用失败：认证过期。
2. **不逐字重试**。读错误 → 属于认证类 → 按路由规则切 lark-shared 处理认证。
3. 认证需要用户交互式登录 → 这是只有用户能做的事 → 停下，给出明确指令（建议用户 `! lark-cli auth login` 在会话内跑），说明卡在哪、做完后我会继续哪一步。
4. 用户完成后：从中断点继续，不从头重来。

### 纪律

- 失败分类后再行动：参数错修参数、环境缺补环境、需要用户的交给用户。
- 停下时的三件套：什么没成 / 为什么 / 接下来怎么办（含用户要做的具体动作）。

## 场景 B：需求中途变化

### 任务

> 原始：「写一篇介绍 MCP 的公众号文章。」
> 中途追加：「对了，加一段和 OpenAI 方案的对比。再顺便把标题做成图。」

### 轨迹

1. **append 不 replace**：需求清单从「①MCP 介绍文」变为「①MCP 介绍文 ②对比段 ③标题图」。原始需求①不因追加而降级。
2. 评估追加项影响：②需要新的检索（OpenAI 方案细节有时效性 → 搜索）；③是新的任务类型 → 路由到配图 Skill。
3. 插入位置判断：②要融入文章结构（不是简单 append 到文末），所以在成稿前处理；③依赖定稿标题，放最后。
4. 交付时**逐项核对**：①文章完整且风格自查过 ②对比段基于检索到的最新信息 ③图已生成并查看。
5. 最终汇报按清单三项逐条给出位置和状态。

### 纪律

- 追加需求最容易挤掉原始需求——清单是防遗漏的唯一可靠机制。
- 追加项一样要走它自己类型的完整流程（检索、Skill、质检），不因为是"顺便"就降标准。

## 场景 C：做到一半发现前提错误

### 任务

> 「修一下部署脚本，它总在上传那步挂。」

### 轨迹

1. 读脚本 + 复现 → 发现挂的不是上传，是上传前的构建产物缺失；"上传挂了"是用户的表面观察。
2. **停止在错误前提上投入**（不继续调上传逻辑）。
3. 向用户重述新理解："真正失败的是构建步骤 X，上传只是被连累的。要修的是 Y。"
4. 因为修 Y 仍在"让部署脚本能跑通"的原始意图内 → 属于同一目标，直接继续修，不需要重新授权；若修 Y 需要动到脚本之外的系统（如 CI 配置），则先确认。

### 纪律

- 用户描述的是**症状**，我要修的是**根因**；两者不一致时对齐理解再继续。
- 判断是否需要重新确认的标准：是否仍在原始意图和影响范围内。
