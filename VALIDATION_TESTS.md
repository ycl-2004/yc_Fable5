# fable-operating-system Skill 验证报告

日期：2026-07-11。方法：用五种不同类型的任务对 Skill 做模拟推演——按 SKILL.md 的五步流程走一遍，记录**可观察的决策摘要**（不是隐藏思维过程），并评估 Skill 是否改变/改善了行为。

---

## 测试 1：简单问答

**输入**：「git 里怎么撤销最近一次 commit 但保留改动？」

| 环节 | 决策摘要 |
|------|----------|
| 理解 | 纯知识问答；交付物 = 直接答案；非时效性 |
| 分类 | 问答类 → 路由表命中「不改文件、不过度排版」 |
| 计划 | 零计划档（planning-framework 三档规则） |
| Skills/工具 | 无（正确地不加载任何东西，包括本 Skill 自身——SKILL.md 的"什么时候不使用"命中） |
| 沟通 | 2 句散文：`git reset --soft HEAD~1`；如果已 push 需要另外处理，一句带过 |
| 质量检查 | 命令正确性（稳定知识，可直接确认） |

**结果**：✅ 通过。
**Skill 的帮助**：主要是**负向约束**——明确了简单任务不启用框架、不排版、不加尾巴。价值在防过度工程。
**遗留问题**：「简单」的边界靠判断，没有硬指标；目前用"是否需要工具调用/是否多步"作代理指标，可接受。

## 测试 2：复杂研究任务

**输入**：「调研 2026 上半年主流模型的 API 定价变化，做个对比，给我一页能看的东西。」

| 环节 | 决策摘要 |
|------|----------|
| 理解 | 检查表第 7 项命中：定价 = 强时效性 → 必须搜索，禁用训练记忆。第 3 项："一页能看的东西" = 交付物形态有歧义（网页/表格/文档） |
| 分类 | 研究类 + 可能的可视化 |
| 计划 | 轻量→显式之间；含 Claude 定价 → **claude-api Skill 阻塞性命中**（提到 LLM 定价不凭记忆答）；产出若含图表 → dataviz 阻塞性命中；若做成网页 → artifact-design 阻塞性命中 |
| 交付物歧义处理 | 按 task-routing §1 第 9 项：可合理默认（Artifact 一页网页最符合"一页能看"）→ 直接做并说明，不提问 |
| 执行顺序 | 并行发多源检索（改进规则 5）→ 交叉验证冲突价格 → claude-api 核对 Claude 侧 → dataviz + artifact-design → 建页 → 发布 |
| 沟通 | 开工一句话；检索完成后简报冲突点；最终消息给链接 + 标注"数据截至 2026-07-11" |
| 质量检查 | 每个价格有来源；时效标注；图表过 dataviz；链接真实可访问 |

**结果**：✅ 通过。
**Skill 的帮助**：显著。这个任务串了**三个易漏的阻塞性 Skill**（claude-api、dataviz、artifact-design），路由表把它们全部挂到了对应环节；旧模式下 dataviz 最可能被漏掉。
**遗留问题**：多个阻塞性 Skill 叠加时的加载顺序没有明文规则（本次按"用到前才读"处理，合理但未成文）→ 已在 skill-tool-map 的纪律中隐含，暂不加规则，观察。

## 测试 3：代码修改任务

**输入**：「给我的 CLI 工具加一个 `--json` 输出选项。」

| 环节 | 决策摘要 |
|------|----------|
| 理解 | 新功能 → **brainstorming 阻塞性命中**（自诊断表 #4 的最高频漏调项，本次被检查表第 6 项显式拦住）→ 先对齐：哪些子命令要支持、错误也输出 JSON 吗、schema 稳定性承诺 |
| 分类 | 代码实现类 |
| 计划 | 轻量计划；TDD 命中：先写 `--json` 输出断言的失败测试 |
| 执行顺序 | 读现有 CLI 参数解析和输出层代码 → brainstorming → 失败测试 → 最小实现 → 全套件 → verify（真的跑一次 `cli --json` 看输出） |
| 沟通 | 汇报：改动文件清单、测试输出、一个真实运行示例 |
| 质量检查 | quality-checks 代码项全过；确认没顺手重构输出层（防跑偏检查第 2 项） |

**结果**：✅ 通过。
**Skill 的帮助**：把 brainstorming→TDD→verify 这条链固化成了默认路径；防跑偏检查拦住了"顺便统一一下现有输出格式"的冲动（记为建议交付）。
**遗留问题**：brainstorming 对"多小的功能算创意工作"仍有灰区（一个 flag 算吗？）。当前决定：**行为可见的新功能一律走**，纯内部重构不走。已写入本报告备查。

## 测试 4：创意设计任务

**输入**：「给我下篇讲"AI 记忆"的文章配三张小黑风格的插图。」

| 环节 | 决策摘要 |
|------|----------|
| 理解 | "小黑" + "文章插图" → 路由到 **ian-xiaohei-illustrations**，不是 yc-ip（两个 IP Skill 的区分点在 IP 形象名） |
| 分类 | 视觉 IP 类；交付物 = 三张成品图文件 |
| 计划 | 轻量：读文章定三个配图位 → 每张一个视觉隐喻 → 生成 → 逐张检查 |
| 依赖 | 需要文章内容——若文章还没写，这是检查表第 5 项的上下文缺口 → 问用户要文章或大纲（这是只有用户能给的输入，提问合理） |
| 质量检查 | 逐张真实查看：构图、中文标注无乱码、风格符合小黑规范（白底手绘 + 红橙蓝批注）；三张放一起看一致性 |
| 沟通 | 交付时说明每张的隐喻选择，支持定向修改 |

**结果**：✅ 通过。
**Skill 的帮助**：IP 路由消歧（yc-ip vs ian-xiaohei-illustrations）在路由表里有明确区分点；"必须真出图 + 必须真查看"两条铁律防住了创意任务的两大失败模式。
**遗留问题**：多张图的批量一致性检查（角色画崩、风格漂移）目前只有"放一起看"一句，可以在未来给配图类任务加一个更具体的批量一致性检查项。

## 测试 5：需求不完整且中途变化的任务

**输入**：「帮我整理一下桌面上那个乱七八糟的项目文件夹。」（未指明哪个、怎么算整理好）
**中途追加**：「哦对，里面有些旧版本文件别删，我还要用。」

| 环节 | 决策摘要 |
|------|----------|
| 理解 | 两个缺口：哪个文件夹（可查——ls 桌面找候选，若多个则问）；"整理好"的标准（部分可推断：按类型/项目归组是惯例默认，但**涉及移动用户文件 = 中风险**） |
| 缺口处理 | 检查表第 9+10 项：先完整盘点（task-workflows §7 铁律：先盘点再动手）→ 给出整理方案让用户过目**再动手**——因为批量移动文件对用户而言接近不可逆（难以凭记忆恢复原状） |
| 中途追加 | append 到需求清单：「旧版本文件不删」→ 立即影响方案：所有删除类动作降级为移入 `_archive/` 暂存 |
| 执行顺序 | 盘点（tree/du）→ 方案确认 → 执行移动 → 前后文件数核对 |
| 质量检查 | 文件总数一致（0 丢失）；追加需求核对：旧版本文件全部在 `_archive/` 可找回 |
| 沟通 | 交付：新结构树 + 每类去向 + "没有删除任何文件"的显式声明 |

**结果**：✅ 通过。
**Skill 的帮助**：这是五个测试里 Skill 价值最大的——需求清单机制（append 不 replace）+ 风险分级（批量移动文件走确认门）+ "不确定的文件进暂存区不删除"的替代方案，全部来自 references。
**遗留问题**：无新增。

## 测试 6：验证模式（v1.1 新增，真实执行）

**输入**：「用 Fable 验证一下上次的自检任务。」

与测试 1-5 不同，这一测不是模拟——审计被真实执行了，产物是 `fable-operating-system/examples/audit-example.md`。

| 环节 | 决策摘要 |
|------|----------|
| 模式识别 | 「用 Fable 验证」→ 验证模式，加载 references/validation-rubric.md |
| 材料收集 | 任务原文（9 项交付物清单）+ 全部产出文件 + VALIDATION_TESTS.md；如实声明缺 `.fable/task.md`（机制当时未建立）导致过程证据只能反推 |
| 七维判定 | A/D/F/G PASS（各附证据引用）；B PASS 附说明；E PASS 附保留（模拟验证证据偏弱）；**C FAIL**——长任务无持久工作文件 |
| 结论 | 有条件通过；整改项「建立工作文件机制」已在 v1.1 落实（templates/task-file.md + 本次升级任务的 `.fable/task.md` 即首个实例） |
| 关键验证点 | 审自己敢不敢给 FAIL——给了，且按现行标准判、不用「规则当时没写」豁免 |

**结果**：✅ 通过。
**证明的能力**：rubric 可操作（每个判定都落到了具体证据）；三档判定能区分「有漏洞可修」和「不能验收」；审计闭环能反哺规则（C 维 FAIL → v1.1 的工作文件机制）。

## 总体结论

1. **五个测试全部通过**，其中测试 2（阻塞性 Skill 串联）和测试 5（需求变化）体现的改善最明显。
2. Skill 的核心价值排序：① 阻塞性 Skill 防漏调 ② 需求清单防遗漏 ③ 简单任务防过度工程 ④ 验证证据门槛。
3. 待完善（已记录，未阻塞发布）：
   - 「简单任务」边界无硬指标（可接受，靠代理指标）
   - 多个阻塞性 Skill 叠加时的加载顺序未成文
   - 批量配图的一致性检查项可以更具体
   - brainstorming 的"创意工作"灰区：采用「行为可见的新功能一律走」的判据
